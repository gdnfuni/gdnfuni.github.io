一、前端主体设计

1.1 设计哲学

风格锚点：深色科技主题（Dark Tech Theme）

本前端采用深色科技主题作为核心设计语言，灵感源自现代开发者工具（Developer Tools）与科学计算软件的视觉风格——如 Mathematica 的计算笔记本界面、MATLAB 的暗色模式、以及 VS Code 的深色主题。这种设计选择基于以下考量：

  1. 功能性优先：计算器是高频率交互的工具型应用，深色背景减少长时间使用的视觉疲劳，同时让彩色数据和运算符在暗色画布上形成高对比度，提升可读性。
  2. 科技感表达：深色背景配合蓝紫色调的发光效果（Glow Effects），天然传递"精密计算""科学严谨"的心理暗示，与产品的计算工具定位高度契合。
  3. 情绪隔离：深色界面能创造一种"专注模式"的心理暗示，帮助用户屏蔽外界干扰，沉浸在计算任务中。

信息密度定位：中等密度（Medium Density）

  • 每个页面聚焦一个核心计算任务，避免功能堆砌
  • 留白（Whitespace）率保持在25%-30%，确保界面呼吸感
  • 通过标签页（Tab）切换组织次级功能，而非一次性展示所有选项

1.2 技术架构

纯原生三件套架构（Vanilla Stack）

经过从 React 到纯 HTML + CSS + JS 的架构重构，最终选择原生三件套方案，核心考量如下：

维度 React 方案 纯原生方案（最终选择）
部署复杂度 需构建工具（Vite/Webpack），输出产物含哈希文件名 无需构建，三文件直接部署
运行时依赖 需加载 React 运行时（~130KB gzip） 零依赖，浏览器原生支持
文件数量 多文件（JSX 组件拆分 + 构建产物） 严格三文件（index.html, style.css, app.js）
服务器配置 需 SPA Fallback 支持 单页但兼容传统服务器
可维护性 组件化，适合大型应用 代码集中，适合中型工具应用
离线可用性 依赖构建产物 可完整保存为本地文件离线使用

最终选择纯原生方案的理由：
- 本应用是工具型单页应用，功能模块间耦合度低但规模可控（~1000行JS）
- 用户核心诉求之一是"简单部署"——三文件可以直接复制到任何Web服务器甚至本地双击打开
- 无需 SPA Fallback 配置，传统静态服务器（如 Nginx、Apache、甚至 Python http.server）即可完美支持
- 零运行时依赖意味着在任何浏览器上都能立即工作，包括老旧设备

├── index.html    # 页面结构 + 6个功能页面的DOM骨架（约180行）

二、后端接口调用设计

2.1 设计视角说明

本章节以双重身份撰写：既是专业程序设计师（负责后端API的设计与实现），又是前端设计师（负责前端如何调用这些API）。这种双重身份确保接口设计既满足后端的技术约束，又符合前端的调用习惯。

2.2 API 调用层设计

核心请求函数

/**
 * 通用API请求函数
 * @param {string} endpoint - API端点路径（如 '/api/calculate'）
 * @param {Object} data - 请求体数据对象
 * @param {Function} callback - 回调函数(error, response)
 */
function apiRequest(endpoint, data, callback) {
    const xhr = new XMLHttpRequest();
    // 构造完整URL：服务器地址 + 端点路径
    xhr.open('POST', AppState.serverUrl + endpoint, true);
    // 声明请求体格式为JSON
    xhr.setRequestHeader('Content-Type', 'application/json');

    xhr.onreadystatechange = function() {
        if (xhr.readyState === 4) {  // 请求完成
            if (xhr.status === 200) {
                const response = JSON.parse(xhr.responseText);
                callback(null, response);
            } else {
                callback(new Error('HTTP ' + xhr.status));
            }
        }
    };

    xhr.onerror = () => callback(new Error('Network Error'));
    xhr.send(JSON.stringify(data));
}

设计决策说明（后端设计师角度）：
- 使用POST方法而非GET：请求体可能包含复杂JSON结构（如矩阵二维数组），URL长度受限且不适合传输复杂数据
- Content-Type: application/json：明确声明请求体格式，后端据此解析
- 异步回调模式：保持非阻塞UI，计算期间用户界面保持响应

设计决策说明（前端设计师角度）：
- 选择XMLHttpRequest而非fetch API：兼容性更好（支持IE11），错误处理更直观
- 全局统一的apiRequest函数：所有API调用走同一通道，便于统一添加日志、错误处理、Loading状态
- 回调函数而非Promise：纯原生JS环境，减少Polyfill依赖

响应处理统一模式

apiRequest('/api/calculate', { expression: '2+3*4' }, (err, res) => {
    if (err) {
        showError('请求失败: ' + err.message);
        return;
    }
    if (!res.success) {
        showError('计算错误: ' + res.error);
        return;
    }
    showResult(res.data.result);
});

三层错误处理：
1. 网络层错误（xhr.onerror）：服务器不可达、超时、CORS拒绝等
2. HTTP层错误（xhr.status !== 200）：404、500等HTTP状态码异常
3. 业务层错误（res.success === false）：表达式语法错误、参数无效等

2.4 错误处理策略

前端错误处理层次

错误类型 检测方式 用户提示 恢复策略
网络不可达 xhr.onerror "无法连接到服务器，请检查网络或服务器地址" 提示检查服务器配置
HTTP 404 status === 404 "请求的资源不存在" 检查API路径是否正确
HTTP 500 status === 500 "服务器内部错误" 稍后重试
业务错误 res.success === false 显示 res.error 内容 根据错误提示修正输入
JSON解析失败 try-catch JSON.parse "服务器返回了无效的数据格式" 检查后端服务是否正常
CORS错误 浏览器控制台 "跨域访问被拒绝，请检查服务器CORS配置" 确认后端已开启CORS

Loading 状态管理

function apiRequestWithLoading(endpoint, data, callback) {
    showLoading(true);  // 显示加载动画（旋转器或进度条）

    apiRequest(endpoint, data, (err, res) => {
        showLoading(false);  // 隐藏加载动画
        callback(err, res);
    });
}

计算密集型请求（如矩阵乘法、大数阶乘）可能耗时较长，前端需显示Loading状态防止用户重复提交。

4.4 触控优化

最小触控目标

所有可点击元素的最小尺寸为 44x44px（Apple Human Interface Guidelines 推荐值）：

.btn {
    min-height: 44px;
    min-width: 44px;
}

@media (min-width: 768px) {
    .btn {
        min-height: 40px;   /* PC端可稍小 */
    }
}

输入框优化

@media (max-width: 767px) {
    .input-field {
        font-size: 16px;     /* 防止iOS自动放大 */
        padding: 14px 16px;
    }
}

重要：iOS Safari 在输入框 font-size < 16px 时会自动放大页面，影响用户体验，因此移动端输入框必须设置 font-size ≥ 16px。

手势支持

  • 左滑/右滑：在标签页之间切换（如排列组合的7种类型）
  • 下拉刷新：历史记录区域下拉刷新（可选）

4.5 性能优化

移动端性能考量

  1. CSS 动画硬件加速：使用 transform 和 opacity 而非 left/top/margin，触发 GPU 加速
    css .page-transition { transition: opacity 200ms ease, transform 200ms ease; will-change: opacity, transform; }

  2. 图片懒加载：如果页面包含复杂图表，使用 loading="lazy" 属性

  3. 减少重绘重排：避免频繁修改 width/height/margin 等触发 Layout 的属性

  4. Debounced 输入：科学计算器的输入框使用防抖（Debounce），避免每次按键都发送请求
    javascript let debounceTimer; inputField.addEventListener('input', () => { clearTimeout(debounceTimer); debounceTimer = setTimeout(() => calculate(), 300); // 300ms防抖 });

4.6 暗色模式与浅色模式

虽然本设计以深色主题为默认，但可以通过 CSS 变量轻松支持浅色模式切换：

五、设计原则总结

5.1 核心设计原则

原则 说明 体现位置
一致性 相同功能的交互方式在所有页面保持一致 所有计算页面统一采用"输入→计算→结果"三段式布局
反馈性 每个操作都有明确的视觉反馈 按钮悬停效果、加载动画、错误提示
容错性 允许用户犯错并提供清晰的恢复路径 输入验证、错误提示、默认值
效率性 高频操作简单快捷 快捷函数按钮、历史记录、预设服务器
美观性 视觉愉悦但不喧宾夺主 深色主题、毛玻璃效果、渐变色

5.4 可访问性（Accessibility）考量

  1. 色彩对比度：所有文字与背景的对比度符合 WCAG AA 标准（至少4.5:1）
  2. 键盘导航：所有交互元素支持 Tab 键聚焦，Enter/Space 键激活
  3. ARIA 标签：关键元素添加 aria-label、role 等属性
  4. 减少动画：支持 prefers-reduced-motion 媒体查询
@media (prefers-reduced-motion: reduce) {
    * {
        animation-duration: 0.01ms !important;
        transition-duration: 0.01ms !important;
    }
}
